from urllib.request import Request, urlopen
from collections import Counter, defaultdict

URL='https://demo.netbox.dev/api/schema/?format=json'
spec=json.loads(urlopen(Request(URL, headers={'User-Agent':'Mozilla/5.0'}), timeout=60).read())

paths=spec.get('paths',{})

op_count=0
methods=('get','post','put','patch','delete')

param_names=Counter();
param_locations=Counter();

list_like=0
bulk_array_request=0
bulk_object_request=0

response_schema_refs=Counter()

# collect common query params per GET
get_query_params=Counter()

for path, item in paths.items():
    if not isinstance(item,dict):
        continue
    for m in methods:
        if m not in item:
            continue
        op=item[m]
        op_count+=1
        # parameters
        for p in op.get('parameters',[]) or []:
            if not isinstance(p,dict):
                continue
            name=p.get('name')
            loc=p.get('in')
            if name:
                param_names[name]+=1
            if loc:
                param_locations[loc]+=1
            if m=='get' and loc=='query' and name:
                get_query_params[name]+=1

        # requestBody analysis
        rb=op.get('requestBody')
        if rb and isinstance(rb,dict):
            content=rb.get('content',{})
            if isinstance(content,dict):
                app=content.get('application/json') or content.get('application/json; charset=utf-8')
                if isinstance(app,dict):
                    sch=app.get('schema')
                    if isinstance(sch,dict):
                        if sch.get('type')=='array':
                            bulk_array_request+=1
                        elif sch.get('type')=='object' or '$ref' in sch or 'oneOf' in sch or 'allOf' in sch:
                            bulk_object_request+=1

        # responses: count $ref to schemas
        for code, resp in (op.get('responses') or {}).items():
            if not isinstance(resp,dict):
                continue
            content=resp.get('content',{})
            if not isinstance(content,dict):
                continue
            app=content.get('application/json')
            if not isinstance(app,dict):
                continue
            sch=app.get('schema')
            if isinstance(sch,dict) and '$ref' in sch:
                response_schema_refs[sch['$ref']]+=1

# infer list-like GET endpoints by presence of limit/offset in params
for path,item in paths.items():
    g=item.get('get') if isinstance(item,dict) else None
    if not isinstance(g,dict):

    q=set()
    for p in g.get('parameters',[]) or []:
        if isinstance(p,dict) and p.get('in')=='query' and p.get('name'):
            q.add(p['name'])
    if 'limit' in q and 'offset' in q:
        list_like+=1

print('Paths:', len(paths), 'Operations:', op_count)

## 1. Identified Schema Patterns

> Source: NetBox demo instance OpenAPI 3.0.3 schema (`info.version`: `4.5.3 (4.5)`). The spec is large: **768** component schemas, **300** paths, **1166** operations.

- **Request/Response schema variants (naming convention)**
  - **Description:** Each “entity” typically has multiple schema variants: response (`Site`), request (`SiteRequest`), writable request (`WritableSiteRequest`), patched request (`PatchedWritableSiteRequest`), plus “brief/nested” reference shapes.
  - **Example from schema:** `Site`, `WritableSiteRequest`, `PatchedWritableSiteRequest`.
  - **Frequency / importance:** Very high (this drives almost every alignment decision).

- **Foreign key relationship encoding differs for read vs write**
  - **Description:** Response often uses an object ref via `allOf: [$ref BriefX]`; requests often use `oneOf: [integer, BriefXRequest]`, commonly with `nullable: true`.
  - **Example from schema:**
    - Read: `Site.region` → `allOf: [$ref BriefRegion]`
    - Write: `WritableSiteRequest.region` → `oneOf: [integer, BriefRegionRequest]`
  - **Frequency / importance:** Very high; `allOf` appears **883** times, `oneOf` appears **749** times.

- **Bulk create + bulk update/delete patterns**
  - **Description:** Many collection endpoints accept either a single object or an array for POST (via `oneOf`). Bulk PATCH/PUT/DELETE typically take an array request body.
  - **Example from schema:** `/api/dcim/sites/` POST schema:
    - `oneOf: [$ref WritableSiteRequest, {type: array, items: $ref WritableSiteRequest}]`
  - **Frequency / importance:** Very high; request bodies with array/object patterns appear broadly (hundreds of ops).

- **Pagination wrapper schemas (Paginated*List)**
  - **Description:** List responses use dedicated wrapper objects containing `count`, `next`, `previous`, `results`.
  - **Example from schema:** `PaginatedSiteList` (results items are `$ref Site`)
  - **Frequency / importance:** High; detected **130** paginated wrapper schemas, matching **130** list-like GET endpoints (`limit`+`offset`).

- **Filtering/query parameter operator suffix patterns**
  - **Description:** List endpoints define many query params with suffix operators like `__gt`, `__lte`, `__empty`, `__n`, plus text variants like `__ic`, `__iregex`, etc. Many are modeled as arrays of values.
  - **Example from schema:** `GET /api/dcim/sites/` has **157** query parameters; e.g. `id__gt` is `type: array` of integers, `created__gte` is `type: array` of `date-time`.
  - **Frequency / importance:** Extremely high (thousands of query params across the spec; query params dominate path metadata).

- **Tagging (cross-cutting many-to-many-ish pattern)**
  - **Description:** Many schemas include `tags` as an array of `NestedTag` (read) or `NestedTagRequest` (write).
  - **Example from schema:** `Site.tags` → `array[NestedTag]`; `WritableSiteRequest.tags` → `array[NestedTagRequest]`
  - **Frequency / importance:** Very high; `tags` exists on **326** schemas.

- **Custom fields (free-form map/object)**
  - **Description:** `custom_fields` is modeled as an object with `additionalProperties: {}` (unconstrained map).
  - **Example from schema:** `Site.custom_fields` → `{type: object, additionalProperties: {}}`
  - **Frequency / importance:** High; `additionalProperties` appears **320** times.

- **Choice fields and “value/label” dual representation**
  - **Description:** Some response schemas represent a choice field as an object with nested `{ value, label }`, while request schemas represent the same choice as a string enum.
  - **Example from schema:** `Site.status` (response) is an object containing `value` enum and `label` enum; `WritableSiteRequest.status` is a string enum.
  - **Frequency / importance:** High; `enum` appears **603** times.

- **Vendor extension: `x-spec-enum-id`**
  - **Description:** Enums often carry `x-spec-enum-id`, reused across request/response variants and filters.
  - **Example from schema:** `Site.status.value` has `x-spec-enum-id`.
  - **Frequency / importance:** High; appears **481** times.

- **Common metadata fields + readOnly**
  - **Description:** Many response schemas include `id`, `url`, `display_url`, `display`, `created`, `last_updated`, often marked `readOnly` (and timestamps often `nullable`).
  - **Example from schema:** `Site.created` → `{format: date-time, readOnly: true, nullable: true}`
  - **Frequency / importance:** Very high; `readOnly` appears **1392** times; `nullable` appears **2651** times.

- **Formats + numeric constraints**
  - **Description:** Heavy use of `format` for `uri`, `date-time`, `int64`, `double`, etc., plus `minimum`/`maximum`.
  - **Example from schema:** `Site.latitude` → `{type: number, format: double, minimum: -90, maximum: 90, nullable: true}`
  - **Frequency / importance:** Very high; `format` appears **1137** times.

- **Binary/file upload fields**
  - **Description:** Some request schemas include `format: binary`.
  - **Example from schema:** `DeviceTypeRequest.front_image` → `{type: string, format: binary, nullable: true}`
  - **Frequency / importance:** Medium (format `binary` appears 8 times).

- **Fixed-length array schema**
  - **Description:** Rare but present: a schema that is itself an array with `minItems == maxItems`.
  - **Example from schema:** `IntegerRange` → array of 2 integers.
  - **Frequency / importance:** Low (only 2 top-level array schemas).

- **Generic relation / polymorphic reference by “type + id”**
  - **Description:** Some schemas use pairs like `assigned_object_type` (string) + `assigned_object_id` (int64), both nullable.
  - **Example from schema:** `IPAddress.assigned_object_type` + `IPAddress.assigned_object_id`
  - **Frequency / importance:** Medium (appears in multiple core resources).

- **Write-only fields**
  - **Description:** Rare fields like passwords are `writeOnly: true`.
  - **Example from schema:** `TokenProvisionRequest.password`
  - **Frequency / importance:** Low (`writeOnly` appears 4 times).

| Pattern | Missing Capability | Impact | Proposed Extension |
|---|---|---|---|
| Request/response variants | Ability to associate an entity with multiple OpenAPI schemas (read vs create vs patch) | Misalignment when validating/auto-discovering fields from schema | Add `api.schemas: { read, create, update, patch }` references (conceptual) |

| Arrays (`tags`, list fields, many-to-many) | Attribute type system lacks arrays + `items` schema | Can’t represent tags or many-to-many from OpenAPI | Extend `NetboxAttributeType` with `array` and `items` definition |
| Map/object fields (`custom_fields`) | No support for `object` + `additionalProperties` | Can’t represent custom fields or other free-form payloads | Add `object` type with `additionalProperties` (boolean or nested schema) |
| Choice “value/label” response shape | No way to represent “choice object” vs “choice string” | Current `status` modeled as scalar enum diverges from response schema | Add `choice: { request: 'string', response: 'value_label_object' }` (conceptual) |
| `format` and numeric width | No `format` support (uri/date-time/int64/double/binary) | Weak alignment; can’t auto-generate correct coercion/validation | Add `format` and optionally `intBits` / `numberFormat` metadata |
| `minLength` | Missing constraint | Incomplete alignment with schema validation | Add `minLength` to attribute definitions |
| Read-only/write-only | Missing | Cannot prevent users setting readOnly fields; can’t model auth fields safely | Add `access: { readOnly, writeOnly }` |

| Filter parameter modeling | Missing | Auto-discovery from schema can’t be stored; filtering features can’t be represented | Add optional `api.filters` section (or per-entity filter map) |
| Generic relation fields | Missing polymorphic relation support | Discovery will treat `*_type/*_id` as plain scalars, losing semantics | Add `relation.kind: generic` and pair-field grouping |

## 4. Conceptual Model Adjustments

These are **conceptual-only** abstractions that would let netbox-model.yaml represent NetBox OpenAPI structures without immediately implementing anything:

1. **Introduce a richer field schema model**
   - Today: `attributes.<name> = { type, required, maxLength, pattern, value, label, nullable, min/max }`
   - Conceptually extend to:
     - `kind: scalar | object | array | relation`
     - `format` (uri/date-time/int64/double/binary/uuid)
     - `minLength`, `minItems`, `maxItems`
     - `readOnly`, `writeOnly`
     - `additionalProperties` for map-like objects
     - `oneOf` support for “id-or-object” write patterns

2. **Separate “domain graph” relationships from “API field relationships”**
   - Current `parent` + `links` encode a diagram placement/ordering dependency graph.
   - The OpenAPI schema encodes relations as *fields* with read/write shapes.
   - Conceptually add an “API-relations” section (or enrich `links`) to capture:
     - target entity type(s)
     - write representation (`integer` PK vs brief object)
     - read representation (`BriefX`/`NestedX`)

3. **Add first-class awareness of request vs response**
   - Since NetBox uses different shapes for read vs write (e.g., `status`, FK fields), the model likely needs:
     - per-operation field schema variants, or
     - a unified field schema with `readShape` and `writeShape`.

4. **Optional API layer metadata (for future auto-discovery)**
   - To align to OpenAPI `paths`, conceptually store:
     - collection path(s) per entity
     - list response wrapper (`Paginated*List`)
     - bulk operation support flags
     - filter/query param definitions (including operator suffix families)

## Clarification on “Patterns” #4 (Filtering Structures) — with examples

What I meant: the OpenAPI schema doesn’t just describe entity *fields*; it also describes a very large, systematic set of **query parameters** for list endpoints (filters/search/pagination). If we ever want “auto-discovery” to include the *API surface* (not just entity schemas), we need to decide how to represent these filters without exploding the model.

### Concrete example: `GET /api/dcim/sites/`
This single endpoint defines **157** query params. Many are “operator families” derived from a base field:

- `name` family (12 operators): `name__empty`, `name__ic`, `name__ie`, `name__iew`, `name__iregex`, `name__isw`, … plus negatives (`__n`, `__nic`, …) and `__regex`.
- `created` family (6–8 operators): `created__gt`, `created__gte`, `created__lt`, `created__lte`, `created__n`, `created__empty` (and some endpoints also have `__after`, `__before`).
- Standard params:
  - `q` (Search string)
  - `ordering` (Sort key)
  - `limit` / `offset` (Pagination)
  - `tag` (array of strings)

### Why this is a modeling question
Across the whole schema:
- There are **130** “list-like” endpoints (have both `limit` and `offset`)
- Across those, there are about **2,080 distinct** operator-suffixed params (e.g. `description__iregex`, `id__lte`, etc.)

So you basically have two representation options (conceptually):
1. **Enumerate every param name** per endpoint (accurate but huge and repetitive).
2. **Store filter “families”** (base field + supported operators + value type), and derive param names as needed.

If your application goal is strictly “diagram → CSV import”, you can likely ignore filters entirely. But for issue #45’s broader “auto-discover by parsing docs”, you’ll eventually hit this question because OpenAPI contains an enormous amount of filter metadata.
